> PTC 是不是要取代 MCP？沙箱里调工具不就是把 MCP 当 API 用吗？LLM 还需要参与执行吗？这篇文章从架构层面厘清两者的关系。

## 问题的起点：传统 Tool Calling 的瓶颈

使用 MCP 构建 Agent 时，标准的工具调用循环是这样的：

```
用户提问 → LLM 决定调工具 A → 执行 → 结果回到上下文
         → LLM 看结果，决定调工具 B → 执行 → 结果回到上下文
         → LLM 看结果，决定调工具 C → 执行 → 结果回到上下文
         → LLM 综合所有结果，生成回答
```

每一次 tool call 都需要一次完整的模型推理，每一次返回的结果都会堆积在上下文窗口中。当工具调用链变长、返回数据量变大时，**token 消耗和延迟会急剧膨胀**。Anthropic 内部测试中，一个多工具工作流的 token 消耗可以从 150K 降到 2K——差距就出在这里。

## PTC 的核心机制

Programmatic Tool Calling（PTC）的思路是把 Agent 循环**拆成两个阶段**：

### 规划阶段（需要 LLM）

Claude 理解用户意图，判断需要哪些工具、怎么组合、中间数据怎么处理，然后把这些决策**一次性编码**成 Python 脚本。

### 执行阶段（不需要 LLM）

脚本在沙箱容器中机械执行。遇到工具调用时暂停，请求发到你的基础设施，结果**直接返回给沙箱而不经过 Claude 的上下文**，脚本继续运行。最终只有处理后的结果进入 Claude 的上下文窗口。

```
Claude 写 Python 脚本 → 沙箱执行
                          ↓
                  遇到 tool call，暂停
                          ↓
                  请求发到你的基础设施
                          ↓
                  结果直接回到沙箱（不经过 Claude）
                          ↓
                  脚本继续执行、处理、过滤
                          ↓
                  只有最终 output 进入 Claude 上下文
```

> [!IMPORTANT]
> 关键设计：**工具返回的原始数据不进入 LLM 上下文**。沙箱是中间商，Claude 只负责"写剧本"和"看结局"。

## PTC 和 MCP 的关系

这是最容易混淆的部分。**PTC 和 MCP 不是竞争关系，而是处于不同的架构层。**

| 维度 | MCP | PTC |
|------|-----|-----|
| **解决什么** | 工具的定义与通信协议 | 工具的调用执行模式 |
| **关注点** | 怎么描述工具、怎么传参、怎么返回 | 怎么编排多个工具、怎么管理上下文 |
| **类比** | HTTP 协议 | 调用 HTTP 接口的编排脚本 |
| **谁定义工具** | 开发者 | 开发者（PTC 不创造新工具） |
| **LLM 的角色** | 每步决策调哪个工具 | 一次性生成编排代码 |

PTC 调用的工具完全可以是 MCP server 暴露的。Anthropic 的前身博客就叫 *"Code Execution with MCP"*。MCP 定义了工具是什么，PTC 改变了工具怎么被调用和结果怎么被处理。

## 一个自然的问题：为什么不直接优化 MCP Tool？

考虑这个场景："查 20 个人的 Q3 报销，找出超预算的人"。最直觉的反应是——**直接在 MCP tool 的服务端实现里加聚合逻辑不就行了？**

没错，如果你能控制工具实现，很多时候应该这么做。PTC 真正有意义的场景更窄，集中在三种情况：

### 1. 你控制不了工具实现

接入的是第三方 SaaS 的 MCP server，它的 `query_expenses` 就是会返回 100 行明细，你没法改它让它帮你聚合。PTC 让 Claude 在沙箱里做中间处理，避免原始数据污染上下文。

### 2. 编排逻辑是动态的

用户问"Q3 超预算的人"和"Q3 花费最多的前三个人"，过滤、排序、聚合方式不同。如果每种组合都写一个 MCP tool，工具数量会爆炸。PTC 让 Claude 根据当前查询动态生成编排代码，调用同一组基础工具但用不同方式组合。

### 3. 需要跨工具拼接

先调工具 A 拿到 ID 列表，再拿 ID 调工具 B，再把 A 和 B 的结果 join 起来计算。这种跨工具编排逻辑没法塞进任何一个 MCP tool 里，除非你造一个"超级工具"把 A 和 B 都包进去——但那不是通用方案。

> [!WARNING]
> 如果过滤和聚合逻辑是固定的、可预见的，**直接在 MCP tool 的服务端实现**是更简单、更可靠的方案。PTC 不应该成为偷懒不优化工具的借口。

## 一个更精确的类比

最初的直觉是"PTC 本质上是 serverless MCP"——工具是预先部署的函数，PTC 是按需即时生成的函数。这个类比在运维模型上很贴切（临时容器、~4.5 分钟过期、无状态），但在职责上需要精确一点：

**PTC 不是 serverless 的工具层，而是 serverless 的编排层。** 工具还是你的 MCP server 在执行，Claude 只是动态生成了调用这些工具的编排脚本，以及脚本里的中间数据处理逻辑。在执行阶段，MCP 确实被当成普通 API 调用了——没有 LLM 参与，纯粹是代码调函数。

## PTC 的适用边界

PTC 的核心前提是：**编排逻辑可以在执行前被确定性代码表达。**

这意味着 Claude 必须在写脚本时就把所有分支判断想清楚。如果执行过程中出现了代码没预见到的情况，沙箱处理不了——不像传统 Agent 循环那样每一步都有 LLM 在看着。

| 场景 | 适合 PTC | 适合传统 Tool Calling |
|------|:--------:|:--------------------:|
| 批量数据查询 + 聚合 | ✅ | |
| 固定流程的多工具编排 | ✅ | |
| 数据过滤和格式转换 | ✅ | |
| 需要语义理解决定下一步的多跳推理 | | ✅ |
| 结果含义模糊需要 LLM 判断的场景 | | ✅ |
| Agentic RAG（基于内容决定检索路径） | | ✅ |

> [!TIP]
> 一个简单的判断标准：如果"根据上一步结果决定下一步"的逻辑可以写成 `if/else` 和 `for` 循环，用 PTC；如果需要 LLM 的语义理解能力来判断，用传统 Tool Calling。

## 实际开发中的决策框架

1. **能在工具服务端解决的，在服务端解决。** 固定的过滤、聚合、格式化逻辑直接写进 MCP tool 实现里，这是最简单可靠的方案。

2. **第三方工具 + 动态编排 = PTC 的甜点场景。** 当你不能改工具实现，且组合方式随用户查询变化时，PTC 才真正发挥价值。

3. **需要每步 LLM 判断力的场景，老老实实用传统循环。** 多跳推理、Agentic RAG、需要根据语义决定下一步的工作流，PTC 处理不了。

4. **混合使用是常态。** 一个复杂 Agent 完全可以在部分工作流用 PTC（批量数据处理），另一部分用传统 Tool Calling（语义推理），两者不矛盾。

## 总结

PTC 和 MCP 的关系可以用一句话概括：**MCP 定义了工具是什么，PTC 改变了工具怎么被编排和执行。** PTC 不创造新工具，不替代 MCP，也不适用于所有场景。它解决的是一个具体问题——当多个工具调用的中间结果不需要 LLM 理解、只需要程序化处理时，把这些"脏活"从上下文窗口转移到沙箱容器中，让 LLM 只看最终结果。

理解这个边界，你就不会在该用 PTC 时浪费 token 做传统循环，也不会在需要 LLM 判断力时误用 PTC 导致执行失败。
